Semantic search

Local-first search that understands meaning, not just keywords. No cloud APIs, no secrets.

Semantic mode

sentence-transformers with all-MiniLM-L6-v2. Understands meaning and finds conceptually related notes.

best quality~80 MB modellocal CPU

TF-IDF mode

scikit-learn TF-IDF vectors. Matches keyword overlap without needing a neural model.

good fallbacklightweightfast

Keyword mode

ripgrep exact string matching via just search. Always available, no install needed.

basiczero installexact match

Git Snapshot Governance

Local memory governance. Preserves vault history as named snapshots. No push by default. No remote by default. No secrets committed.

How it works

  1. Initialize Git locally (no remote created)
  2. Run safety checks: forbidden files, .gitignore coverage
  3. Preview what would be committed with dry-run
  4. Generate a snapshot manifest describing system state
  5. Create a named commit with all allowed files
  6. Blocked if forbidden files (secrets, keys, cache) are detected

Will NOT

  • Push to any remote
  • Create remotes
  • Commit .env, secrets, credentials, or private keys
  • Commit machine-local state (workspace, cache, venv)
  • Rewrite history
  • Delete files
  • Run as a background daemon
  • Assume GitHub exists

Commands

python scripts/william.py git-status
python scripts/william.py git-init
python scripts/william.py snapshot --dry-run
python scripts/william.py snapshot --message "Baseline WilliamOS v1"

Manifest: 91_GitGovernance/SNAPSHOT_MANIFEST.md (committed with each snapshot)

Forbidden file detection

The snapshot engine blocks commits if any forbidden files are detected: .env, *.pem, *.key, id_rsa, credentials.*, secrets.*, and more. Filenames containing secret, token, password, or api_key are flagged as suspicious. Forbidden files block the commit. Suspicious files are reported but do not block.

Restore workflow

git log --oneline                          # find snapshot
git checkout <hash> -- .                   # restore files
python scripts/william.py check            # verify integrity
git bundle create backup.bundle --all      # portable backup

Restore Drill / Disaster Recovery

A backup is not trusted until restore is tested. Extracts to a safe temporary folder, runs health checks, and proves the archive is operational.

How it works

  1. Verify archive checksum (SHA-256)
  2. Extract to a temporary folder outside the live repo
  3. Check required files and directories exist
  4. Check no forbidden files leaked into the archive
  5. Verify Git history is present
  6. Run health checks: check, git-status, backup-status, cockpit-status
  7. Generate restore drill report and manifest
  8. Clean up temporary folder (unless --keep)

Will NOT

  • Overwrite the live WilliamOS folder
  • Delete live files
  • Move restored files over live files
  • Push to any remote
  • Create cloud sync
  • Modify source notes
  • Rewrite Git history
  • Claim success unless checks pass

Commands

python scripts/william.py restore-status
python scripts/william.py restore-drill --archive path/to/archive.zip --dest C:\temp\restore-test
python scripts/william.py restore-drill --archive path/to/archive.zip --dest C:\temp\restore-test --keep
python scripts/william.py restore-drill --latest --dest C:\temp\restore-test
python scripts/william.py restore-manifest

Reports: 93_RestoreDrill/reports/ • Manifest: 93_RestoreDrill/RESTORE_MANIFEST.md

Confidence levels

HIGH — all checks pass, checksum verified, all health checks pass.
MEDIUM — required files present, Git history present, some health checks fail.
LOW — missing required files or Git history.

Private Remote Strategy

Remote is optional. No remote by default. No push by default. William chooses the protection strategy. Privacy and control come first.

Recommended order

  1. Local Git snapshot — version history on this machine
  2. External encrypted drive — verified archive on separate hardware
  3. Restore drill — proof that backup is restorable
  4. Encrypted off-machine archive — geographic redundancy
  5. Optional: private Git remote — only if explicitly approved
  6. Optional: Obsidian Sync — convenience, not sole backup

Will NOT

  • Create any remote
  • Push to any service
  • Upload notes to cloud
  • Store tokens or credentials
  • Configure sync daemons
  • Choose a provider automatically
  • Require internet access
  • Enable background sync

Official Acceptance Assistant

One-item-at-a-time controlled acceptance. Copy a reviewed draft into its official folder — only with explicit --confirm. No batch, no auto-accept, no overwrite.

How it works

  1. Plan — accept-plan validates draft, checks duplicates, verifies sections, shows exact action
  2. Review — read the plan, fix any blockers, confirm destination is correct
  3. Accept — accept-draft with --confirm copies the draft to official folder
  4. Log — every acceptance is logged with timestamp, source, and destination
  5. Snapshot — run snapshot after accepting to preserve the change

Copy-by-default — original draft stays in source folder.

Will NOT

  • Auto-accept or batch-accept drafts
  • Move files without --confirm
  • Overwrite existing official notes
  • Delete original drafts
  • Accept from unknown source folders
  • Accept into unknown destination folders
  • Push to remote

Commands

python scripts/william.py accept-status
python scripts/william.py accept-plan --draft "80_DoctrinePromotion/drafts/Doctrine - Example.md"
python scripts/william.py accept-plan --draft "path/to/draft.md" --dest "WilliamOS/03_Doctrine/"
python scripts/william.py accept-draft --draft "path/to/draft.md" --dest "WilliamOS/03_Doctrine/" --confirm
python scripts/william.py accept-log

Post-Acceptance Closure

After accepting a draft, run closure to refresh queues, check system health, and get a snapshot recommendation. No commit, no backup, no push.

How it works

  1. Accept — accept a draft via accept-draft --confirm
  2. Closure — post-acceptance refreshes queues, cockpit, runs check
  3. Review — read the closure report, verify no warnings
  4. Snapshot — use the recommended snapshot message

Optional: add --refresh-cortex to also rebuild the cortex map.

Will NOT

  • Accept, move, or delete any files
  • Auto-commit or auto-backup
  • Push to remote
  • Modify source notes

Production Readiness Gate

Final pre-release checkpoint. Aggregates all subsystem checks into a single PASS/FAIL verdict.

9-Point Gate

  1. Global governance check
  2. Runtime smoke (29 commands)
  3. Restore runtime proof
  4. Schema validation (36 schemas)
  5. Command registry (86 commands)
  6. Backup archive exists
  7. Git safety (no remote)
  8. Required docs present
  9. No forbidden files

Verdicts

  • PASS — all 9 checks pass. Safe to tag.
  • WARN — blocking checks pass but non-critical smoke failures. Tag with caution.
  • FAIL — blocking check failed. Do not tag.

Will NOT

  • Auto-tag or auto-release
  • Modify source notes
  • Push, sync, or create remotes

Commands

python scripts/william.py production-status
python scripts/william.py production-readiness
python scripts/william.py restore-runtime-proof --latest --dest <path>

Architecture

One process serves everything. FastAPI backend on port 8420 serves the API (/api/*) and the pre-built React frontend from control-center/frontend/dist/. No Node.js process at runtime. The operator agent is deterministic and rule-based — same state produces same recommendations. Every command goes through the safety gate before execution. 8-point smoke test validates all endpoints.
