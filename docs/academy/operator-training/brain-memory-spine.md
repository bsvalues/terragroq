# Brain Memory Spine

Brain Memory is WilliamOS continuity under governance. It helps the Primary see
facts, decisions, procedures, patterns, contradictions, stale items, review
queues, sensitivity posture, and authority state without treating remembered
context as permission.

## What Memory Can Represent

- Fact Memory: a sourced claim with confidence, sensitivity, evidence, and review state.
- Decision Memory: a prior owner or system decision with context, options, outcome, and safety flags.
- Procedure Memory: a repeatable procedure with authority, validation, evidence, and stop conditions.
- Pattern Memory: a repeated operating lesson with examples, limits, confidence, and stale triggers.
- Contradiction Memory: a conflict that blocks trust until reviewed.
- Stale Memory: context that may no longer match current evidence or `origin/main`.
- Review Queue Item: a proposed memory record that needs evidence or owner review.

## What Memory Cannot Do In This Lane

- It cannot write, edit, promote, delete, archive, or persist memory.
- It cannot retrieve runtime context, create embeddings, store vectors, or add RAG.
- It cannot ingest files, GitHub, email, chat logs, backups, Docker, PACS, or county data.
- It cannot execute commands, run Work Orders, activate Brain Council runtime, or activate Hermes/MCP/workers.
- It cannot authorize production writes, DB/schema changes, auth changes, env/package changes, or Vercel changes.

## How To Read Memory Safely

Use memory as context to inspect, not as authority to act. The safe sequence is:

1. Check source and evidence.
2. Check confidence and sensitivity.
3. Check stale or contradiction state.
4. Check authority state and owner decision gates.
5. Convert any action into a bounded Work Order before mutation.

If memory conflicts with current evidence, current evidence wins. If memory
conflicts with the newest owner instruction, the newest owner instruction wins.
