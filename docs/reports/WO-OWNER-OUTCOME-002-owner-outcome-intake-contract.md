# WO-OWNER-OUTCOME-002 — Owner Outcome Intake Contract

## Status

`PENDING_AFTER_WO_OWNER_OUTCOME_001`

## Contract

An owner outcome is a plain-language statement of the result William wants. It is not required to contain a program ID, goal ID, loop ID, Work Order ID, branch name, command, or authorization keyword.

WilliamOS/Codex must translate the outcome into a bounded execution packet containing:

- one WilliamOS-native program identity;
- one goal and loop identity;
- allowed and blocked scope;
- a risk ceiling;
- dependency and evidence requirements;
- a useful Work Order chain;
- stop conditions limited to genuine authority or safety walls;
- a continuation target that does not return routine planning to William.

Frustration with inactivity or an explicit direction to continue useful work is valid owner direction. The system must not demand a magic word when the intended bounded outcome is already clear.

## Safety

Ambiguous requests involving protected external projects, production, secrets, credentials, paid spending, runtime activation, destructive action, or R2+ scope still require a concise owner decision packet. Routine WilliamOS-native R0/R1 continuation does not.
